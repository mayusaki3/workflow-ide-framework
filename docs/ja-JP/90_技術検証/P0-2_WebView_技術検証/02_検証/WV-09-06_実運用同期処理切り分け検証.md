# WV-09-06 実運用同期処理切り分け検証

## 目的

WV-09-05完了時点で残存している主因候補である実運用同期処理について切り分けを行い、Linux応答停止の発生条件を特定する。

## 背景

WV-09-05までの検証により、以下の要素は単独では応答停止を発生させないことを確認した。

- GTK Host Window
- Native Child Window
- GtkFixed階層
- WebKitGTK
- WebView配置制御
- WebView表示制御
- eframe / winit共存

ソースコード調査の結果、実運用時に継続実行される処理は主に以下であることを確認した。

- GTKイベントポンプ
- Native Surface位置同期
- Visibility同期

そのため、これらを主因候補として切り分けを実施する。

## 検証方針

実装上存在する同期処理を対象として切り分けを実施する。

まず対象同期処理を停止した状態を確認し、その後、単独有効状態で応答停止再現有無を確認する。

## 検証構成

- GDK_BACKEND=x11
- eframe
- egui
- GTK Host Window
- Native Child Window
- WebKitGTK

## 検証項目

| 検証番号        | 項目                 | 期待結果     |
| ----------- | ------------------ | -------- |
| WV-09-06-01 | GTKイベントポンプ停止 | 実施完了 |
| WV-09-06-02 | GTKイベントポンプ最小化 | 実施完了 |
| WV-09-06-03 | Visibility同期停止 | 未実施 |
| WV-09-06-04 | GTKイベントポンプ単独有効 | 未実施 |
| WV-09-06-05 | Native Surface位置同期単独有効 | 未実施 |
| WV-09-06-06 | Visibility同期単独有効 | 未実施 |

## 判定基準

応答停止を再現した場合

- 当該同期処理を主因候補とする

応答停止を再現しない場合

- 次の同期処理へ進む

## 検証結果

### WV-09-06-01 GTKイベントポンプ停止

実施完了。

観測結果は以下の通り。

- ビルド成功
- 起動成功
- gtk::init 成功
- WebView build_gtk 成功
- Native Child Window同期成功
- WebView set_bounds 成功
- WebView set_visible(true) 成功
- GTKイベントポンプは skip ログを出力し、実処理されない
- Window は表示されない
- 応答なしは発生しない

### WV-09-06-02 GTKイベントポンプ最小化

実施完了。

観測結果は以下の通り。

- ビルド成功
- 起動成功
- gtk::init 成功
- 初期化時の GTKイベントポンプ実行成功
- WebView build_gtk 成功
- WebView生成直後の GTKイベントポンプ実行成功
- Native Child Window同期成功
- WebView set_bounds 成功
- WebView set_visible(true) 成功
- 同期フェーズの GTKイベントポンプは skip ログを出力し、実処理されない
- デスクトップ左上に Window が表示される
- Dockでタブをドラッグすると Window が消える
- Window消失後は再表示されない
- 応答なしは発生しない

## 判定

WV-09-06-01 は不合格とする。

GTKイベントポンプ停止状態では応答なしは発生しなかったが、Window が表示されないため、表示成立状態での原因切り分けとしては成立しない。

WV-09-06-02 は不合格とする。

初期化時および WebView生成直後に限定して GTKイベントポンプを実行すると、Window の初期表示は成立するが、Dock操作後の表示追従は成立しない。

ただし、応答なしは発生しなかった。

## 知見

GTKイベントポンプは Window 表示成立に必要である。

また、初期化時および WebView生成直後のみ GTKイベントポンプを実行した場合、Window の初期表示は成立する。

一方で、Dock操作後に Window が消え、再表示されないことから、Dock追従中または Visibility同期中にも GTKイベントポンプが必要である可能性が高い。

WV-09-06-01 および WV-09-06-02 の両方で応答なしが発生しなかったため、応答なし現象は GTKイベントポンプを含む継続同期フェーズに依存している可能性が高い。

## 結論

GTKイベントポンプは表示成立に必要であり、主因候補から除外できない。

ただし、GTKイベントポンプを停止または最小化すると応答なしは発生しない。

現時点では、応答なしの主因候補は以下の組み合わせへ絞り込む。

- 継続同期フェーズの GTKイベントポンプ
- Native Surface位置同期
- Visibility同期

## 次工程

WV-09-06-03 として、GTKイベントポンプを Dock追従同期後に限定して実行する検証を実施する。

初期化時、WebView生成直後、Native Surface位置同期後のみ GTKイベントポンプを実行し、Visibility同期後は実行しない状態で、Window表示、Dock追従、応答なし再現有無を確認する。
