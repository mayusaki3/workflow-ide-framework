# WV-09-06 実運用同期処理切り分け検証

## 目的

WV-09-05完了時点で残存している主因候補である実運用同期処理について切り分けを行い、Linux応答停止の発生条件を特定する。

## 背景

WV-09-05までの検証により、以下の要素は単独では応答停止を発生させないことを確認した。

- GTK Host Window
- Native Child Window
- GtkFixed階層
- WebKitGTK
- WebView配置制御
- WebView表示制御
- eframe / winit共存

ソースコード調査の結果、実運用時に継続実行される処理は主に以下であることを確認した。

- GTKイベントポンプ
- Native Surface位置同期
- Visibility同期

そのため、これらを主因候補として切り分けを実施する。

## 検証方針

実装上存在する同期処理を対象として切り分けを実施する。

まず対象同期処理を停止した状態を確認し、その後、単独有効状態で応答停止再現有無を確認する。

## 検証構成

- GDK_BACKEND=x11
- eframe
- egui
- GTK Host Window
- Native Child Window
- WebKitGTK

## 検証項目

| 検証番号        | 項目                 | 期待結果     |
| ----------- | ------------------ | -------- |
| WV-09-06-01 | GTKイベントポンプ停止 | 実施完了 |
| WV-09-06-02 | Native Surface位置同期停止 | 未実施 |
| WV-09-06-03 | Visibility同期停止 | 未実施 |
| WV-09-06-04 | GTKイベントポンプ単独有効 | 未実施 |
| WV-09-06-05 | Native Surface位置同期単独有効 | 未実施 |
| WV-09-06-06 | Visibility同期単独有効 | 未実施 |

## 判定基準

応答停止を再現した場合

- 当該同期処理を主因候補とする

応答停止を再現しない場合

- 次の同期処理へ進む

## 検証結果

### WV-09-06-01 GTKイベントポンプ停止

実施完了。

観測結果は以下の通り。

- ビルド成功
- 起動成功
- gtk::init 成功
- WebView build_gtk 成功
- Native Child Window同期成功
- WebView set_bounds 成功
- WebView set_visible(true) 成功
- GTKイベントポンプは skip ログを出力し、実処理されない
- Window は表示されない
- 応答なしは発生しない

## 判定

WV-09-06-01 は不合格とする。

GTKイベントポンプ停止状態では応答なしは発生しなかったが、Window が表示されないため、表示成立状態での原因切り分けとしては成立しない。

## 知見

GTKイベントポンプは Window 表示成立に必要である。

一方で、GTKイベントポンプを停止すると応答なしが発生しないことから、応答なし現象は GTKイベントポンプそのもの、または GTKイベントポンプと Native Surface / WebView 操作の組み合わせに依存している可能性が高い。

確認ログ上、GTKイベントポンプ停止中でも以下の処理は成功している。

- WebView build_gtk
- Native Child Window同期
- WebView set_bounds
- WebView set_visible(true)

そのため、単純な WebView生成、Native Child Window同期、set_bounds、set_visible の単発成功だけでは表示成立を保証しない。

## 結論

GTKイベントポンプ停止は応答なしを抑止するが、Window 表示も成立しない。

GTKイベントポンプは主因候補から除外できない。

次工程では、GTKイベントポンプを完全停止するのではなく、表示成立に必要な最小イベントポンプ条件を確認する。

## 次工程

WV-09-06-02 として、GTKイベントポンプ最小化検証を実施する。

GTKイベントポンプを初期化直後および WebView生成直後に限定して実行し、継続同期中の GTKイベントポンプを停止した状態で、Window表示と応答なし再現有無を確認する。
