# WV-10 WV07再現条件差分分析

## 目的

WV-07で Linux応答なしが発生していた時点と現在の develop ブランチとの差分を分析し、Linux版 WebView 実装で応答なしが再現する条件を確認する。

本検証の目的は、応答なしの詳細原因調査を継続することではない。

WV-09までの切り分けで、個別要素および同期処理単体では応答なしを再現できなかったため、本検証では現在の検証用実装を退避したうえで、想定される最終実装を作成し、その実装で応答なしが再現するかを確認する。

## 背景

WV-09では、WV-08で除外できなかった要素を段階的に再導入し、実運用同期処理を含めて切り分けを行った。

しかし、WV-09-06までの範囲では応答なしを再現できず、同期処理単体を主因として特定できなかった。

また、Windows版 WebView 対応後、Linux版で主に変更されている対象は `linux_webview.rs` である。

このため、WV-07時点の再現条件を過去実装から詳細に復元するよりも、現在の検証コードを退避し、Windows版と同等の責務を持つ Linux 最終実装候補を作成して動作確認する方が、次工程判断に適している。

## 前提

以下を前提とする。

- 対象ブランチは `develop` とする。
- HLDocS v0.6.1-preview を正本とする。
- 現在の `linux_webview.rs` は WV-09-06-05 検証用実装である。
- `app.rs` 側の差分は主要因として扱わない。
- 応答なしの深掘り調査ではなく、最終実装候補の成立性確認を優先する。
- 現在の検証コードは削除せず、退避して復旧可能にする。

## 検証方針

本検証は、以下の二段階で実施する。

### 1. 差分確認

WV-07実施時点から現在の develop までの差分を確認し、Linux版 WebView 実装で主要な変更対象が `linux_webview.rs` に集中していることを確認する。

比較対象は主に以下とする。

- `src/platform/linux_webview.rs`
- `src/app.rs`
- Dock矩形取得処理
- Native Surface同期処理
- GTKイベントポンプ処理
- Visibility同期処理
- WebView生成・配置処理

### 2. 最終実装候補による再現確認

現在の検証用 `linux_webview.rs` を退避したうえで、想定される Linux版 WebView 最終実装候補を作成する。

最終実装候補では以下を満たす。

- WV番号に依存する検証コードを除去する。
- Windows版 WebView 実装と同等の責務に整理する。
- Linux固有の GTK / WebKitGTK / wry 処理のみを `linux_webview.rs` に閉じ込める。
- WebView生成、Native Surface同期、Visibility同期、GTKイベント処理を実運用前提の構造に整理する。
- 過剰な検証ログを削除し、必要最小限の診断ログにする。
- 応答なしが再現するかを Linux 環境で確認する。

## 判定基準

### 合格

以下をすべて満たす場合、WV-10を合格とする。

- 現在の検証用実装を退避できている。
- Linux最終実装候補を作成できている。
- Linux環境で起動確認できている。
- 応答なしが発生しない、または発生条件を最終実装候補上で明確に確認できている。
- 次工程で実装継続または追加切り分けを判断できる。

### 条件付き合格

以下の場合、WV-10を条件付き合格とする。

- 最終実装候補で応答なしは発生しないが、WebViewの表示・追従・Visibilityのいずれかに制限が残る。
- 最終実装候補で応答なしが再現するが、再現条件が実装構造上の特定箇所に絞り込める。

### 保留

以下の場合、WV-10を保留とする。

- 最終実装候補を作成できない。
- Linux環境で実行確認できない。
- 応答なし以外の環境要因により判定できない。

## 検証結果

### WV-10-07 GTK WindowType 変更確認

`gtk::WindowType::Popup` を `gtk::WindowType::Toplevel` に変更した結果、GTK Window が最前面になるだけでは応答なしにならなくなった。

一方で、GTK Window 上にマウスカーソルを置く、または GTK Window 上でマウスを動かし続けると、約5秒後に応答なしが発生した。

このため、`Popup` 固有の問題だけではなく、表示された GTK Window と入力・イベント処理の相互作用が関与していると判断した。

### WV-10-08 `WebView::set_bounds()` 停止確認

`WebView::set_bounds()` の呼び出しを停止しても、GTK Window 上にマウスカーソルがある場合の応答なしは再現した。

このため、`WebView::set_bounds()` は主因ではないと判断した。

### WV-10-09 `GtkFixed::set_sensitive(false)` 確認

`GtkFixed::set_sensitive(false)` により GTK 配下の通常入力イベントを抑制しても、応答なしは再現した。

このため、`GtkFixed` の通常入力イベント処理は主因ではないと判断した。

### WV-10-10 GTK / WebKitGTK 完全無効化確認

GTK Window、GtkFixed、WebKitGTK WebView、GTKイベント処理をすべて生成・実行しない構成では、応答なしは発生しなかった。

このため、`eframe` / `egui` / `winit` / Dock処理単体では応答なしは発生せず、GTK / WebKitGTK を生成する構成に依存して応答なしが発生していると判断した。

### WV-10-11 X11親子化方式確認

`WINIT_UNIX_BACKEND=x11` を指定して実行し、eframe / winit 側の親 Window を X11 Window として取得し、GTK Window を `XReparentWindow()` により子 Window 化する方式を検証した。

結果として、`RawWindowHandle` は `Wayland` を返し、X11 Window は取得できなかった。

また、`gdk_x11_window_get_xid()` は `GDK_IS_X11_WINDOW` アサーションで失敗し、GTK Window も X11 Window ではなかった。

`XOpenDisplay()` は成功したが、親 Window・子 Window の XID が取得できず、X11 親子化は成立しなかった。

GTK Window を生成する構成では、従来どおり Linux の応答なしが発生した。

## 現在状態

WV-10は完了。

WV-10の検証により、Linux の応答なしは GTK / WebKitGTK を生成する構成に依存して発生することを確認した。

`WINIT_UNIX_BACKEND=x11` 指定下でも実際には Wayland Window として動作しており、X11 親子化方式は成立しなかった。

そのため、GTK / WebKitGTK をホスト Window とする方式の検証は終了し、GTK を使用しない Linux WebView 実装方式の検討へ移行する。

## 次工程

WV-12として、GTK を使用しない Linux WebView 実装方式を検討する。

候補方式は、WRY の GTK 非依存利用可否、Chromium / CEF 系、外部プロセス WebView、または Linux 専用の別方式とする。
