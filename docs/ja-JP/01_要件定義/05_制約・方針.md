# 制約・方針

Workflow IDE Framework の開発にあたり、以下の制約・方針を定める。

## 1. 特定技術に依存しすぎない

特定AI・特定Runtime専用にしない。

## 2. 拡張可能であること

後からNode・Runtime・Pluginを追加可能とする。

## 3. 再利用可能であること

他プロジェクトでも利用可能な構造とする。

## 4. 人間が理解しやすいこと

ノーコード/ローコード利用者でも扱いやすい構造とする。

## 5. 実行中状態を扱えること

静的定義だけではなく、
実行中Workflowも管理可能とする。

## 6. AI連携を前提とすること

AI利用を後付けではなく、
初期設計段階から考慮する。

## 7. 複数環境で動作可能であること

ローカル・サーバー・ブラウザなど、
複数実行環境への対応を考慮する。

## 8. Framework本体を肥大化させすぎない

用途固有機能はPluginや外部Runtimeへ分離可能とする。

## 9. 長期運用可能であること

保守・テスト・ドキュメント整備・拡張を継続可能な構造とする。

## 10. 導入が容易であること

利用者向けにインストーラを用意し、
関連パッケージなどの導入支援機能を持つこと。

## 11. Runtime 分離

UI と Runtime は process 分離可能な構成を考慮する。

また、必要に応じて別マシン・別OS・別Runtime環境へ分離可能な構成を考慮する。

## 12. GPU Viewport

MuJoCo や VRM preview などを埋め込み表示可能な GPU Viewport を考慮する。

また、リアルタイム描画・3D表示・シミュレーション表示を扱える構造を考慮する。

## 13. 外部Runtime連携

Framework内部のみで完結せず、
外部Runtime・外部プロセス・外部機器との連携を考慮する。

## 14. Plugin による機能分離

用途固有機能は可能な限り Plugin や外部モジュールとして分離し、
Framework本体との疎結合を維持する。

## 15. AI Runtime 切り替え

OpenAI など特定AIサービスへの固定依存を避け、
AI Runtime を切り替え可能な構造を考慮する。

## 16. Workflow と Runtime の分離

Workflow定義とRuntime実装を分離し、
同一Workflowを複数Runtimeで利用可能な構造を考慮する。

## 17. リアルタイム処理への配慮

ロボット制御・VR・シミュレーションなど、
リアルタイム性を必要とする用途への利用を考慮する。

## 18. 状態同期を考慮すること

複数Runtime・UI・外部システム間での状態同期を考慮する。

## 19. オフライン利用を考慮すること

クラウド接続前提とせず、
ローカル環境のみでも利用可能な構成を考慮する。

## 20. 開発環境と実行環境を分離可能であること

開発者向け環境と、
エンドユーザー向け実行環境を分離可能な構成を考慮する。
